3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про шаблони (патерни) проєктування
     Шаблони або патерни проєктування є фундаментальним інструментом у сучасній інженерії програмного забезпечення. Історично концепція патернів була запозичена з будівельної архітектури та адаптована для програмування так званою «Бандою чотирьох» (Gang of Four). Їх головне призначення полягає у забезпеченні розробників стандартизованими, перевіреними часом та практикою рішеннями для найбільш типових архітектурних проблем. Використання шаблонів дозволяє уникнути багаторазового розв'язання ідентичних задач, зменшує кількість архітектурних помилок на ранніх етапах розробки та формує єдиний професійний словник, завдяки якому інженери можуть ефективно комунікувати. Згідно з класифікацією GoF, усі двадцять три класичні патерни концептуально поділяються на три великі групи: породжувальні, структурні та поведінкові. Породжувальні патерни абстрагують та жорстко контролюють процес інстанціювання об'єктів, приховуючи логіку їх створення від клієнтського коду, що робить систему незалежною від способів ініціалізації її компонентів. Структурні патерни визначають ефективні способи композиції класів та об'єктів у значно більші підсистеми, забезпечуючи при цьому збереження гнучкості та продуктивності навіть при зміні окремих фрагментів коду. Поведінкові патерни, до яких належить об'єкт даного дослідження, регламентують складні алгоритми взаємодії, способи комунікації та чіткий розподіл обов'язків між об'єктами під час безпосереднього виконання програми. Основною областю застосування всіх перелічених шаблонів є об'єктно-орієнтоване проєктування, розробка високонавантажених серверних додатків, створення прикладних програмних інтерфейсів, побудова архітектури розподілених систем та глибокий рефакторинг існуючих монолітних кодових баз.


3.2 Опис обраного патерна
     Назва патерна: Iterator (Ітератор)
     Тип шаблона (патерна): Поведінковий.
     Основне призначення: Надання універсального механізму для послідовного доступу до елементів будь-якого складеного об'єкта або колекції даних, категорично не розкриваючи деталей його внутрішнього базового представлення.
     Передумови застовуння:
- Наявність об'єктивної необхідності приховати складну структуру даних (дерева, графи, хеш-таблиці) від клієнтського коду за вимоги забезпечення зручного лінійного обходу.
- Потреба у забезпеченні підтримки кількох паралельних або різних алгоритмів обходу однієї колекції (наприклад, у глибину та в ширину).
Переваги:
- Суворе дотримання принципу єдиної відповідальності (SRP) шляхом інкапсуляції логіки обходу в окремому об'єкті.
- Можливість одночасного та незалежного обходу колекції декількома різними ітераторами (кожен зберігає власний стан).
Недоліки:
- Ускладнення загальної архітектури системи через збільшення кількості класів та абстракцій.
- Надлишкова оптимізація при використанні для тривіальних структур даних


3.3 Структура шаблона (патерна)
     
     Рисунок 3.1 – UML діаграма шаблона
     Опис структурних частин:
1. Iterator (Абстрактний ітератор): Інтерфейс, який декларує стандартизований контракт для обходу. Включає методи getNext() (для отримання поточного елемента і переходу до наступного) та hasMore() (для перевірки наявності нерозглянутих елементів).
2. IterableCollection (Абстрактна колекція): Інтерфейс, що репрезентує колекцію даних і містить єдиний фабричний метод createIterator(), який повертає об'єкт типу Iterator.
3. ConcreteCollection (Конкретна колекція): Клас, що імплементує інтерфейс IterableCollection. Містить фактичну структуру даних та реалізує фабричний метод для створення відповідного ітератора.
4. ConcreteIterator (Конкретний ітератор): Клас, що імплементує інтерфейс Iterator. Інкапсулює поточний стан обходу (iterationState) та містить асоціативне посилання на об'єкт ConcreteCollection.
Опис взаємодії об’єктів:
     Клієнтський код (Client) взаємодіє зі структурами даних виключно через абстрактні інтерфейси (Iterator та IterableCollection), що гарантує слабке зв'язування. Колекція створює екземпляр ітератора та передає йому посилання на себе. Під час виклику методів обходу (наприклад, getNext()), конкретний ітератор звертається безпосередньо до внутрішніх елементів конкретної колекції.
     На UML-діаграмі (рис. 3.1) взаємодія відображена чіткими структурними зв'язками. Client залежить від обох абстрактних інтерфейсів. IterableCollection має зв'язок залежності зі створюваним Iterator. ConcreteIterator утворює асоціацію з ConcreteCollection для забезпечення прямого доступу до даних колекції під час ініціалізації та подальшого обходу.
3.4 Реалізація шаблона (патерна)
     Основні етапи:
1. Створення базових інтерфейсів: У контексті C# стандартом індустрії є використання вбудованих узагальнених інтерфейсів IEnumerator<T> (роль Ітератора) та IEnumerable<T> (роль Агрегату). Спочатку створюється базова модель даних — клас Employee, що інкапсулює відомості про працівника і виступає інформаційним вузлом для перебору.
2. Реалізація конкретних класів:
- Реалізація Конкретного Ітератора: Створюється клас DepartmentEnumerator, що імплементує IEnumerator<Employee>. Він реалізує властивість Current, методи MoveNext() (зсув вказівника із безпечною перевіркою виходу за межі) та Reset(), а також зберігає стан — змінну _position.
- Реалізація Конкретного Агрегату: Створюється клас Department, який імплементує IEnumerable<Employee>. Клас інкапсулює масив Employee[] та реалізує фабричний метод GetEnumerator(), який інстанціює та повертає об'єкт DepartmentEnumerator.
3. Демонстрація використання: У клієнтському коді (головному методі) створюється екземпляр відділу (Department) та наповнюється даними. Демонстрація виконується двома шляхами:
- Ручний обхід: Явний виклик GetEnumerator() та використання циклу while для безпосередньої роботи з механікою патерна.
- Синтаксичний обхід: Використання циклу foreach, який під капотом працює з інтерфейсами IEnumerable та IEnumerator, приховуючи від клієнта деталі зберігання даних у класичному масиві.

1. public class DepartmentIterator : Iterator
2. {
3.     private Department _collection;
4.     private int _position = -1;
5.     private bool _reverse = false;
6. 
7.     public DepartmentIterator(Department collection, bool reverse = false)
8.     {
9.         _collection = collection;
10.         _reverse = reverse;
11.         if (reverse)
12.             _position = collection.GetItems().Count;
13.     }
14. 
15.     public override object Current() => _collection.GetItems()[_position];
16. 
17.     public override int Key() => _position;
18. 
19.     public override bool MoveNext()
20.     {
21.         int updatedPosition = _position + (_reverse ? -1 : 1);
22.         if (updatedPosition >= 0 && updatedPosition < _collection.GetItems().Count)
23.         {
24.             _position = updatedPosition;
25.             return true;
26.         }
27. 
28.         return false;
29.     }
30. 
31.     public override void Reset() =>
32.         _position = _reverse ? _collection.GetItems().Count - 1 : 0;
33. }

3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про систему
     Назва системи: Pinterest
     Розробник: Pinterest, Inc.
     Призначення системи: Pinterest — це глобальна візуальна пошукова система та платформа соціальних медіа, призначена для пошуку, збереження та організації ідей (рецептів, інтер'єрів, стилю тощо) у вигляді зображень та коротких відео ("пінів"), які групуються на тематичних "дошках".
     Область застосування та користувачі: Платформа розрахована на масового споживача та бізнес. Система обслуговує понад 600 мільйонів активних користувачів щомісяця.
     Основні функції: Завантаження та обробка медіаконтенту, семантичний та візуальний пошук зображень (Computer Vision), побудова персоналізованої інтелектуальної стрічки рекомендацій (Smart Feed) у реальному часі, підтримка графових соціальних зв'язків (підписки на користувачів та дошки).
3.2 Архітектурний стиль системи
     Архітектура Pinterest еволюціонувала від монолітного вебдодатка (побудованого на Python/Django) до гібридного архітектурного стилю, що поєднує Сервіс-орієнтовану архітектуру (SOA / Мікросервіси) та Подієво-орієнтовану архітектуру (Event-Driven Architecture — EDA).
     Монолітна архітектура не могла забезпечити швидкість розгортання нових функцій та масштабування. Перехід до мікросервісів дозволив розподілити бізнес-логіку на сотні незалежних компонентів (сервіс профілів, сервіс індексації, сервіс стрічки). Це дало змогу незалежно масштабувати найбільш навантажені вузли та використовувати оптимальні мови програмування для різних задач.
     Оскільки користувачі генерують величезну кількість контенту, обробляти його синхронно неможливо. Подієво-орієнтований підхід із використанням брокерів повідомлень дозволяє системі швидко відповідати клієнту, переносячи "важкі" задачі у фоновий асинхронний режим.
3.3 Основні компоненти системи
     Система має яскраво виражену багаторівневу (N-tier) топологію, розміщену в хмарному середовищі Amazon Web Services (AWS).
     - Інтерфейс користувача (Клієнтський рівень): Включає мобільні додатки для iOS та Android, а також вебдодаток (SPA), побудований на базі React. Клієнти відповідають за рендеринг UI, локальне кешування контенту та відправку телеметрії.
     - Серверна частина (Edge Layer & API Gateway): Точка входу в систему. Реалізована на базі проксі-серверів Envoy та HAProxy. Цей рівень виконує термінацію SSL/TLS-з'єднань, первинне балансування навантаження, захист від DDoS-атак, лімітування запитів (rate limiting) та маршрутизацію трафіку до внутрішніх мікросервісів.
     - Рівень бізнес-логіки (Compute Layer): Набір stateless-мікросервісів (сервісів без стану). Основні з них: Pin Service (управління життєвим циклом зображень), Smart Feed Service (агрегація та ранжування контенту для головної сторінки), Zen (Graph Engine) — сервіс обчислення соціального графа.
     - Рівень зберігання даних (Storage Layer): Найбільш складна частина архітектури, розділена за типами даних: Amazon S3 (використовується виключно як об'єктне сховище для оригіналів пінів та їхніх оптимізованих копій), Sharded MySQL (основна транзакційна база даних. Кластер складається з тисяч серверів. Для масштабування використовується підхід шардування — розподіл даних між серверами на основі згенерованого 64-бітного ідентифікатора (ID), де частина бітів вказує на номер фізичного шарда), HBase / Cassandra (колонкові NoSQL бази даних, оптимізовані для швидкого читання графових структур).


3.4 Взаємодія компонентів
     Взаємодія в системі відбувається за двома основними патернами: синхронним (для читання) та асинхронним (для запису/обробки).
     Клієнт надсилає HTTP-запит до API Gateway. Запит маршрутизується до сервісу Smart Feed. Сервіс звертається до кластеру Redis (механізм кешування). Якщо стрічка користувача знайдена (Cache Hit), вона миттєво повертається. Якщо ні (Cache Miss) - сервіс надсилає внутрішні gRPC-запити до баз MySQL та HBase, агрегує дані, пропускає їх через ML-модель ранжування, зберігає результат у Redis і повертає клієнту.
     При завантаженні зображення API-сервер синхронно зберігає файл в S3, робить легкий запис метаданих в MySQL і миттєво повертає клієнту код 200 OK. Одночасно сервер формує повідомлення і кладе його в топік Apache Kafka. Незалежні воркер-сервіси (Workers) читають це повідомлення, створюють thumbnails, аналізують зображення та оновлюють пошуковий індекс Elasticsearch.
     Для забезпечення консистентності даних між MySQL та кешем або графовими базами використовується технологія CDC (Change Data Capture). Спеціальні процеси читають бінарні логи (binlog) MySQL і транслюють зміни у Kafka, звідки вони застосовуються до інших сховищ.

Рисунок 3.1 – Діаграма послідовності
     На діаграмі (рис. 3.1) відображено асинхронний пайплайн створення контенту, де клієнт взаємодіє з API, після чого ініціюється відкладена обробка через брокер повідомлень Kafka, що дозволяє уникнути тайм-аутів з боку клієнтського додатка.

     Рисунок 3.2 – UML діаграма компонентів
     Діаграма (рис. 3.2) ілюструє структуру багаторівневої архітектури, демонструючи чітке розділення між рівнем маршрутизації (API Gateway), мікросервісами, шаром кешування та рівнем шардованих баз даних.
3.5 Масштабованість та надійність
     Архітектура спроєктована за принципами високої доступності (High Availability) та лінійного горизонтального масштабування.
     Механізми балансування навантаження реалізовані на трьох рівнях: глобальне балансування через DNS (GeoDNS направляє користувача в найближчий дата-центр AWS), апаратні/програмні балансувальники на вході в мережу (AWS ELB), та внутрішнє балансування через Service Mesh для рівномірного розподілу трафіку між мікросервісами.
     Система спроєктована з урахуванням того, що апаратні збої є неминучими. Бази даних мають архітектуру Master-Slave з автоматичним перемиканням (failover). Шардування обмежує "радіус ураження" (blast radius): якщо один шард падає, це впливає лише на малу частку відсотка користувачів.
     Є критичним елементом надійності. Відмова кластера Redis може призвести до каскадного падіння (cascading failure) баз даних MySQL через різке зростання кількості прямих запитів. Тому кеші також є кластеризованими та реплікованими.
     Дані реплікуються як в межах однієї зони доступності (Availability Zone), так і між різними географічними регіонами (Cross-Region Replication) для аварійного відновлення.
     Усі мікросервіси запаковані в Docker-контейнери та керуються системами оркестрації (Kubernetes). Це дозволяє автоматично збільшувати кількість екземплярів сервісу (Auto-Scaling) під час пікових навантажень (наприклад, у вихідні або на свята).
3.6 Приклади програмної реалізації
     У додатку В наведено фрагменти програмного коду мовою C#, що демонструють практичну реалізацію ключових архітектурних патернів, застосованих у досліджуваній системі, а також запити до ШІ.
     Перший фрагмент коду ілюструє оптимізацію процесу отримання даних (на прикладі завантаження профілю користувача) з використанням in-memory сховища Redis та реляційної бази MySQL. Алгоритм роботи мікросервісу побудований за стратегією "Cache-Aside" (відкладене кешування), що є критично важливим для забезпечення низької затримки (low latency).
     Під час надходження запиту сервіс формує унікальний ключ і звертається безпосередньо до кластера Redis. Якщо дані присутні (сценарій Cache Hit), вони десеріалізуються з формату JSON і миттєво повертаються клієнту. Це забезпечує час відгуку на рівні 1–2 мілісекунд.
     У випадку відсутності даних у кеші (сценарій Cache Miss), система виконує SQL-запит до відповідного шарда MySQL (у прикладі використовується мікро-ORM Dapper для швидкого мапінгу даних).
     Отримані з основної бази дані не лише повертаються користувачу, а й асинхронно записуються назад у Redis. При цьому обов'язково встановлюється час життя запису (TTL — Time To Live, наприклад, 1 година). Це гарантує, що кеш не буде нескінченно розростатися і зберігатиме лише актуальні дані.
     Архітектурна перевага цього підходу полягає у зниженні навантаження на дискову підсистему баз даних до 90%, що дозволяє системі витримувати різкі стрибки трафіку.
     Другий приклад демонструє застосування подієво-орієнтованої архітектури (EDA) під час операції запису — завантаження нового медіаконтенту (піна). Замість виконання повного, ресурсомісткого циклу обробки зображення в межах одного HTTP-запиту, API-сервіс виконує лише функцію маршрутизації та делегування.
     Після швидкого збереження оригіналу файлу в об'єктне сховище (Amazon S3), сервіс генерує об'єкт події. Цей об'єкт містить лише найнеобхідніші метадані: тип події, ідентифікатор нового об'єкта, ID автора та шлях до файлу в хмарі.
     Для зв'язку з брокером використовується клієнт Kafka (бібліотека Confluent.Kafka). Конфігурація налаштована на баланс між швидкістю та надійністю: параметр підтвердження (Acks = Acks.Leader) вимагає відповіді лише від головного вузла брокера, що мінімізує мережеві затримки.
     Серіалізована JSON-подія відправляється у спеціалізований топік (наприклад, events_pin_uploaded), після чого мікросервіс одразу звільняє потік і повертає клієнту статус успішного виконання операції.
     Архітектурна цінність цього фрагмента полягає в ізоляції процесів. Поки клієнт бачить повідомлення "Пін створено", інші незалежні мікросервіси (Worker Services) асинхронно зчитують цю подію з Kafka і починають фонову обробку: генерацію мініатюр, оновлення пошукових індексів Elasticsearch та розпізнавання об'єктів на зображенні за допомогою моделей машинного навчання. Це запобігає блокуванню потоків API-шлюзу та забезпечує масштабованість системи алгоритмів.

В.1 Приклад №1: Cache-Aside Pattern
GitHub репозиторій:
https://github.com/NureZomaHlib/apz-pzpi-23-10-zoma-hlib/blob/main/Pract2/pzpi-23-10-zoma-hlib-pract2/code-examples-pract2.cs
1. public class UserProfile
2. {
3.     public long Id { get; set; }
4.     public string Username { get; set; }
5.     public string Bio { get; set; }
6. }
7. 
8. public class UserProfileService
9. {
10.     private readonly IDatabase _cache;
11. 
12.     private readonly string _dbConnectionString;
13. 
14.     private readonly TimeSpan _cacheExpiration = TimeSpan.FromHours(1);
15. 
16.     public UserProfileService(IConnectionMultiplexer redis, string dbConnectionString)
17.     {
18.         _cache = redis.GetDatabase();
19.         _dbConnectionString = dbConnectionString;
20.     }
21. 
22.     public async Task<UserProfile> GetProfileAsync(long userId)
23.     {
24.         string cacheKey = $"user:profile:{userId}";
25. 
26.         var cachedData = await _cache.StringGetAsync(cacheKey);
27.         if (cachedData.HasValue)
28.         {
29.             return JsonSerializer.Deserialize<UserProfile>(cachedData!);
30.         }
31. 
32.         using var connection = new MySqlConnection(_dbConnectionString);
33.         string sql = "SELECT Id, Username, Bio FROM Users WHERE Id = @Id";
34.         var profile = await connection.QuerySingleOrDefaultAsync<UserProfile>(sql, new { Id = userId });
35. 
36.         if (profile != null)
37.         {
38.             var serialized = JsonSerializer.Serialize(profile);
39.             await _cache.StringSetAsync(cacheKey, serialized, _cacheExpiration);
40.         }
41. 
42.         return profile;
43.     }
44. }
